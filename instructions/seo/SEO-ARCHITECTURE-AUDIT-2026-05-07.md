## Schema Findings

### What's working well

- All 7 ticker analyses (Prada, Dickson, Jacobio, Yadea, Alibaba, Haier, Tencent Music) carry the locked May 2026 SEO pattern: `Article` + `image` + `inLanguage` + `wordCount` + `articleSection` + `isPartOf` + expanded `keywords` + `mainEntityOfPage` + `about`.
- `BreadcrumbList` is auto-generated by `build.js` from the in-body breadcrumb. Verified working: emits for articles only, skips hubs and homepage as designed.
- `Person` schema on `/about` (Marc, with `knowsAbout`).
- `WebSite` schema on the homepage with author reference.
- `<html lang="en" dir="ltr">` is hard-set in `build.js`. No hreflang needed.

### Issue 1. HSI article missed the May 6 SEO upgrade. (HIGH)

| Field | Status |
|---|---|

| `inLanguage` | MISSING |

### Issue 2. Sector hubs have no JSON-LD. (HIGH × 7)

**Files**: `luxury.html`, `biotech.html`, `technology.html`, `electric-vehicles.html`, `consumer-discretionary.html`, `special-situations.html`, `market-thesis.html`.

Every ticker analysis declares:

```json
"isPartOf": {
  "@type": "CollectionPage",
  "@id": "https://trading852.com/analyses/luxury",
  "name": "Luxury — HKEX Research"
}
```

But the target page at `/analyses/luxury` (and the 6 sister hubs) emits no schema at all. Google sees `@id` references pointing to pages with no matching entity. The reference is dangling.

Fix: emit a `CollectionPage` JSON-LD on each hub, including an `ItemList` of the published articles. This closes the `@id` loop and gives Google a structured signal for hub pages, which is exactly what the recently published `seo-architecture` rule on multi-type `@graph` patterns is designed for.

Template (apply to each hub, swapping the article list):

```json
{
  "@context": "https://schema.org",
  "@type": "CollectionPage",
  "@id": "https://trading852.com/analyses/luxury",
  "name": "Luxury — HKEX Research",
  "description": "Independent research on luxury sector companies listed on HKEX. Documented valuation discounts, brand analysis, identifiable catalysts.",
  "url": "https://trading852.com/analyses/luxury",
  "inLanguage": "en",
  "isPartOf": {
    "@type": "WebSite",
    "@id": "https://trading852.com/#website"
  },
  "mainEntity": {
    "@type": "ItemList",
    "itemListOrder": "https://schema.org/ItemListOrderDescending",
    "numberOfItems": 2,
    "itemListElement": [
      {
        "@type": "ListItem",
        "position": 1,
        "url": "https://trading852.com/analyses/1913-prada",
        "name": "EPS Down 74%. Revenue Up 9%. One of These Numbers Is a Distraction."
      },
      {
        "@type": "ListItem",
        "position": 2,
        "url": "https://trading852.com/analyses/0113-dickson-concepts",
        "name": "The Market Is Paying You HKD 375 Million to Buy This Company."
      }
    ]
  }
}
```

Note the em dash in `"Luxury — HKEX Research"` is already in the article-side `isPartOf` references and would need to match. If you prefer to ban em dashes there too, change both sides to `"Luxury · HKEX Research"` or `"Luxury Sector HKEX Research"` in one sweep.

### Issue 3. No standalone Organization entity sitewide. (MEDIUM)

**File**: `src/_partials/head.html`.

Articles reference `"publisher": { "@type": "Organization", "name": "Trading852", "url": "https://trading852.com" }` inline. There is no canonical `Organization` entity at a stable `@id`. Result: every article repeats a thin Organization stub. Google has no central place to attach the brand identity (logo, social profiles, contact).

Fix: add a sitewide Organization JSON-LD block to `head.html`, then have `Article.publisher` (and `WebSite.publisher`, `CollectionPage.isPartOf`) reference its `@id` instead of redeclaring it. Save as a constant in `build.js` so it ships on every page.

```json
{
  "@context": "https://schema.org",
  "@type": "Organization",
  "@id": "https://trading852.com/#organization",
  "name": "Trading852",
  "url": "https://trading852.com",
  "logo": {
    "@type": "ImageObject",
    "url": "https://trading852.com/assets/og-image.png",
    "width": 1200,
    "height": 630
  },
  "sameAs": [
    "TBD: confirm X/Twitter, LinkedIn, or other public profiles"
  ]
}
```

Then in articles: `"publisher": { "@id": "https://trading852.com/#organization" }`.

If there are no public social profiles to list, drop `sameAs` entirely. Better than a placeholder.

## Sitemap Findings

**File**: `src/static/sitemap.xml`. 20 URLs total, well below the 50k split threshold. No sitemap index needed.

### Issue 4. Stale `<lastmod>` on 4 URLs. (HIGH)

The May 6 README rule is explicit: "Sitemap and JSON-LD freshness must agree". Currently they don't:

| URL | Sitemap `lastmod` | Reality | JSON-LD `dateModified` | Verdict |
|---|---|---|---|---|
| `/analyses/1167-jacobio` | 2026-05-06 | Verdict corrected to MONITOR on May 7 (originally labelled CONVICTION at Apr 14 publication in error). Article body now carries a Correction notice block. | 2026-05-06 | Bump both to **2026-05-07** |
| `/scorecard` | 2026-04-20 | Major rewrite May 7: post-stop live price, trailing-stop tier methodology block | n/a | Bump to **2026-05-07** |
| `/about` | 2026-04-11 | Expanded 253→600 words on Apr 27 (changelog) | n/a | Bump to **2026-04-27** |
| `/` (homepage) | 2026-05-06 | Recent Analyses card eyebrows + biotech category card updated May 7 | n/a | Bump to **2026-05-07** |

Stale `lastmod` on a page that genuinely changed suppresses Google re-crawl frequency. The May 6 rule was correctly applied to 7 articles; these 4 just got missed.

### Issue 5. HSI article `lastmod` matches its missing JSON-LD upgrade. (LOW)

`/analyses/hsi-35-year-trendline` shows `lastmod 2026-04-11`. Internally consistent with the un-upgraded JSON-LD (Issue 1). Once Issue 1 is fixed, bump this `lastmod` and the JSON-LD `dateModified` to the same upgrade date.

### Issue 6. `<priority>` and `<changefreq>` are dead weight. (LOW)

Both tags are ignored by Google as of 2017. Sitemap currently uses them on every URL. Cosmetic only; remove if you want a leaner file. Not a fix, but worth noting since the README's own validation table flags it as `Info` severity.
